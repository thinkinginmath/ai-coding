# AI 编程挑战

本目录包含用于评估**使用 AI 编程工具时的人类判断力**的编程挑战，而不仅仅是 AI 能力本身。

## 设计理念

传统的编程测试通常衡量的是候选人能否产出可运行的代码。有了 AI 工具，几乎每个人都能产出可运行的代码。这些挑战则衡量：

1. **权衡分析** - 能否对架构决策进行推理？
2. **边界情况识别** - 能否发现 AI 遗漏的问题？
3. **渐进复杂性处理** - 当需求演变时能否重构？
4. **诚实的自我评估** - 知道自己不知道什么？

## 挑战题目

**类型**：权衡决策

**考察技能**：
- 架构推理

**评分分布**：
- 40% 设计文档（权衡、替代方案）

### 2. 多阶段：电商购物车

**类型**：渐进复杂性

**考察技能**：
- 渐进式构建
- 处理不断演变的需求
- 重构决策
- 并发推理

**评分分布**：
- 第一阶段是获得任何分数的必要条件
- 第二阶段是获得 B 或更高分数的必要条件

## 如何组织考试

### 建议时间

| 挑战 | 建议时间 |
|------|----------|
| 通知系统 | 2-3 小时 |
| 电商购物车（所有阶段） | 3-4 小时 |

### 环境要求

候选人需要：
- Node.js 18+
- npm 或 yarn
- 他们喜欢的 AI 编程工具

### 评估建议

**关注点**：
- 提问的质量（澄清问题、边界情况）
- 设计文档中的权衡推理
- 对局限性的诚实态度
- 代码质量和推理质量

**警示信号**：
- 没有设计推理，只有代码
- 忽略需求中提到的边界情况
- 简单部分过度工程，复杂部分工程不足
- 无法解释自己的代码

### 评分标准概要

| 等级 | 特征 |
|------|------|
| A | 深入的权衡分析，捕捉边界情况，诚实地承认局限性，代码整洁 |
| B | 方案可工作，架构合理，处理主要场景 |
| C | 基本方案可工作，推理薄弱，遗漏边界情况 |
| D | 部分方案，没有有意义的设计讨论 |
